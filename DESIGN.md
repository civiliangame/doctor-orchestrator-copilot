# Design: Doctor Orchestrator Copilot (DOC) — Hackathon Build

Generated by /office-hours on 2026-07-18
Status: APPROVED
Mode: Builder (hackathon)

## Problem Statement

A doctor ends a visit knowing exactly what the next visit is for — "check the chest pain hasn't progressed, escalate to meds if it radiates" — but that intent lives in a free-text note that the nurse, the imaging tech, and the next specialist never really see. Each department re-collects context from scratch, asks generic questions, and the original intent leaks away between stations.

DOC carries the doctor's intent across the gap. At the end of visit N, the doctor states the purpose, goals, and guardrails for visit N+1 in plain text. DOC turns that into a routed itinerary. When the patient returns, every department activates DOC as an ambient copilot: it listens to the live conversation, suggests the next question or action based on the full patient context plus the previous doctor's goals, and keeps a single evolving chart current for the next specialist.

One-liner for judges: **"Scribes document the visit. DOC orchestrates it."**

## What Makes This Cool

The "whoa" is live mid-conversation orchestration: a role-played patient says "actually it's been spreading to my left arm," and within seconds the nurse's screen shows *"Dr. Zhang's guardrail: radiating pain → escalate. Ask about numbness and shortness of breath now. Cardiology consult recommended before imaging."* The agent visibly heard the room and changed the plan — using intent set by a different doctor days earlier.

**Contradiction detection is a headline feature.** On every patient turn, DOC cross-checks what was just said against everything it already knows — the seeded record, last visit's notes, and earlier statements in this same session. When the patient says "I've been taking the aspirin every day" but the chart shows they reported stopping it at intake twenty minutes ago, DOC flags it live with the exact conflicting sources and a suggested probe. Humans miss these because no single staff member holds the whole context; DOC holds all of it.

Three demo beats, one agent, one patient journey (chest-pain follow-up):

1. **Beat D (setup):** Doctor types next-visit intent in plain text → DOC generates the routed itinerary with goals and guardrails, doctor edits and confirms.
2. **Beat A (climax):** Live nurse-intake session. Real voice, real diarization, suggestions, guardrail alerts, and contradiction flags appear per patient turn.
3. **Beat C (payoff):** Imaging station uploads a CT slice / lab panel; DOC's multimodal read appends findings to the chart; final screen shows the accumulated chart the specialist inherits.

Deprioritized: computer-use on the EHR (Beat B) — post-hackathon idea, not a stretch goal within the 24 hours.

## Constraints

- **Solo builder, ~24 hours** of build time.
- **Stack:** FastAPI backend on localhost orchestrating everything; React (Vite) frontend; SQLite for the patient store.
- **Voice:** Soniox STT via LiveKit (the Soniox API key is provisioned through LiveKit, so LiveKit stays in the stack). Speaker diarization, grounded by the staff member's scripted greeting ("Hi, I'm Doctor Zhang, how are you doing today?") — the first speaker is always staff. No typed *transcript* fallback; operator cues are allowed (see risk checkpoint in Next Steps).
- **LiveKit deployment:** LiveKit Cloud (free tier) — no self-hosting. FastAPI mints room tokens at `POST /session/start`; a LiveKit Agents worker (Python, `livekit-agents` + Soniox STT plugin) joins each room server-side, receives diarized transcript events, and forwards them to the orchestrator in-process.
- **LLM:** Claude API. Fast model (Sonnet tier) for per-turn suggestion calls where latency matters; same API's multimodal for image reads.
- **Demo:** ~3 minutes, projector + one laptop, internet required (Claude, Soniox, and LiveKit Cloud are all cloud services) — mobile hotspot as backup.

## Premises

1. **Demo-first ruthlessness.** One patient, one scripted journey (chest-pain follow-up: nurse intake → imaging → doctor), fully seeded synthetic data. Anything judges can't see in 3 minutes gets faked or cut.
2. **Computer-use EHR is cut from the 24h plan entirely.** The "mock EHR" is a seeded patient-record store with a simple UI panel; DOC reads/writes it via API. Computer-use is a post-hackathon idea.
3. **The novelty on display is cross-visit intent.** The demo must visibly show DOC using *last visit's* goals and guardrails to change what happens in the room. If judges miss this, we've demoed a generic chatbot.
4. **Vision beat is staged, not general.** 2–3 pre-selected images (CT slice, lab panel) with known findings, one multimodal call each, framed as "flagging for the specialist," never "diagnosing."
5. **Voice is Soniox via LiveKit with greeting-grounded diarization.** The scripted staff greeting at session start anchors speaker labels; the orchestrator fires on patient end-of-turn.
6. **Zero clinical-validity burden.** Synthetic data only, visible disclaimer in the UI. Judges score workflow and orchestration, not medical accuracy.

## Approaches Considered

### Approach A: FastAPI monolith with per-turn orchestrator (CHOSEN)

One FastAPI app: LiveKit/Soniox transcript flows in; on every completed **patient turn** the backend calls Claude with (patient record + last-visit goals/guardrails + station context + transcript so far) and pushes suggestion cards, guardrail alerts, contradiction flags, and chart updates over WebSocket to the React UI. SQLite patient store. Vision = upload endpoint → Claude multimodal. Itinerary = plain-text intent → Claude → editable checklist.

- Effort: M | Risk: Low
- Pros: fewest moving parts; every piece is a well-trodden pattern; suggestion cards project well on stage.
- Cons: chart state is implicit in tables; the handoff view needs deliberate UI work.

### Approach B: Tool-use agent + FHIR-lite structured chart

Claude as a true agent with tools (read_chart, update_chart, check_guardrails, suggest_question, flag_finding), per-station state machine, typed chart schema.

- Effort: L | Risk: Medium
- Pros: closest to the real product; tool-call log is an impressive "watch it think" panel.
- Cons: schema + tool plumbing can eat 6+ hours before anything is demoable — dangerous solo in 24h.

### Approach C: "The chart IS the agent" — one living Visit Brief document

All state in a single structured brief; the same agent re-reads and edits it at every station; UI renders the evolving document.

- Effort: S/M | Risk: Low-Med
- Pros: radically simple state; the demo artifact is inherent.
- Cons: whole-document rewrites can flicker or drop detail; needs section-level patching discipline.

## Recommended Approach

**Approach A.** The orchestrator fires **after every completed patient turn** — never on a timer or utterance batch. This keeps suggestions tightly coupled to what was just said, which is the property that makes Beat A land.

### Architecture

```text
Browser (React/Vite)
  ├─ Doctor Planning view  ──REST──►  FastAPI (localhost:8000)
  ├─ Station view (nurse/imaging/doctor)                │
  │    ├─ mic ──► LiveKit Cloud room ◄── LiveKit Agents worker (Soniox STT plugin)
  │    │                                        │ diarized transcript events
  │    └─ ◄──WebSocket── suggestion cards / guardrail alerts / contradictions / chart deltas
  └─ Chart Handoff view ──REST──►
                                    FastAPI
                                      ├─ Token minting (/session/start → LiveKit room token)
                                      ├─ Orchestrator (per patient turn → Claude, structured JSON out)
                                      ├─ Vision endpoint (image → Claude multimodal → findings)
                                      ├─ Itinerary generator (intent text → Claude → plan)
                                      └─ SQLite (patients, visits, goals, guardrails, stations,
                                                 transcript, suggestions, chart_entries)
```

**Speaker-to-role mapping (diarization is advisory, alternation is authoritative):** after the scripted staff greeting anchors "first speaker = staff," turns are assumed to **strictly alternate** staff/patient for the rest of the session — the demo script is written that way on purpose. Diarization labels are used to confirm; when they disagree with alternation (label swap, phantom third speaker from cross-talk or room noise), alternation wins. This makes a mid-demo diarization glitch invisible instead of fatal.

**Patient end-of-turn detection:** fire the orchestrator when a *final* transcript segment attributed to the patient is followed by **900ms with no further patient-attributed final segments**. The debounce collapses the common case of one spoken turn arriving as 2–3 final fragments a few hundred ms apart.

**Tick concurrency:** at most **one in-flight Claude call per session**. If a new patient turn completes while a call is in flight, cancel the pending call and re-fire with the fuller transcript. No queuing, no parallel ticks — this prevents duplicate suggestions and out-of-order chart writes.

**Orchestrator tick (core loop):**

1. Debounced, diarization-confirmed patient end-of-turn arrives.
2. Backend assembles: hand-written patient summary (part of the seed data — no runtime compaction call) + visit N goals/guardrails + current station role + session transcript (last 20 turns) + suggestions already shown (to avoid repeats).
3. One Claude call returns structured JSON (schemas below).
4. Persist to SQLite; broadcast over WebSocket; UI renders cards/toasts/chart deltas.

**Orchestrator output schemas** (all five arrays, kept deliberately small):

```json
{
  "suggestions":      [{ "text": "Ask about numbness in the left arm", "reason": "Guardrail: radiating pain", "priority": "high|normal" }],
  "guardrail_alerts": [{ "guardrail_id": 3, "triggered_by": "patient: 'spreading to my left arm'", "action": "Escalate: cardiology consult before imaging" }],
  "chart_updates":    [{ "category": "symptom|vital|finding|note|contradiction", "text": "Chest pain now radiating to left arm, onset ~4 days ago" }],
  "handoff_notes":    [{ "for_station": "imaging|doctor", "text": "Prioritize views of the upper-left thoracic region" }],
  "contradictions":   [{ "statement": "patient: 'I've been taking the aspirin every day'", "conflicts_with": "intake transcript 12:03 — 'I stopped the aspirin two weeks ago' (and visit-N note: daily aspirin prescribed)", "severity": "high|note", "suggested_probe": "Ask when they last actually took it and why they stopped" }]
}
```

Contradictions render as a distinct amber card (visually separate from suggestion cards), citing both conflicting sources verbatim, and are also appended to `chart_entries` with `category = contradiction` so they survive into the handoff view. The orchestrator prompt explicitly instructs: compare this turn against (a) the seeded record, (b) last visit's notes/goals, and (c) every earlier statement in this session's transcript.

`chart_updates` append to a `chart_entries` table `(id, visit_id, station, ts, category, text)` — **append-only, no merging or upserts**. The Chart Handoff view is a render of `chart_entries` grouped by station, plus `handoff_notes` filtered to the viewing station. That's the entire chart data model.

**Latency budget for Beat A:** end-of-turn → card on screen must feel < ~3s, and this is **measured, not assumed**: the first task of the orchestrator block (H9) is timing one real tick end-to-end (Soniox endpoint → card render). If it exceeds 3s: trim transcript context to the last 10 turns, cap output tokens harder, and stream the response so suggestion cards render as they parse.

**Station role prompting:** the same orchestrator, parameterized by a station enum (nurse / imaging / doctor). Only the **nurse persona gets tuned** — it's the only station with a live voice session in the demo. Imaging and doctor enum values exist but stay stock.

## Open Questions

- Judging format: live on your laptop, or recorded video? (Determines how much to invest in a backup recording.)
- Room audio: will a single laptop mic pick up two role-players reliably, or bring an external mic?
- Does the hackathon have API-credit or model restrictions?

## Success Criteria

- The full 3-minute demo (Beat D → A → C) runs end-to-end **twice in a row** with no intervention by hour 21 (scripted operator cues from the H9 fallback, if that de-scope was taken, don't count as intervention).
- At least one guardrail alert fires live from *spoken* input during Beat A.
- At least one contradiction is caught live during Beat A: the demo script plants one (patient contradicts either the chart or their own earlier statement), and DOC's amber card cites both conflicting sources.
- The chart handoff view visibly accumulates content from ≥2 stations.
- A backup screen recording of one clean run exists before demo day.

## Distribution Plan

Localhost demo only. No CI/CD, no deployment — deliberately out of scope for the hackathon. If it wins and becomes a project, deployment is a follow-up design.

## Next Steps

1. **H0–2:** Scaffold FastAPI + Vite React app; SQLite models; seed the synthetic patient (prior visit notes, meds, history, **hand-written patient summary** used verbatim in the orchestrator prompt) and the chest-pain journey data.
2. **H2–4:** Beat D — planning view: intent text → Claude → editable itinerary + goals/guardrails, saved to store.
3. **H4–9:** Voice pipeline — LiveKit Cloud project + token endpoint, Agents worker with Soniox STT plugin, greeting-grounded speaker labeling with the strict-alternation rule, debounced patient end-of-turn events landing in FastAPI. **This is the highest-risk block; do it early, not last.**
   **H9 hard checkpoint:** if diarized end-of-turn isn't firing reliably by H9, de-scope — ship single-speaker STT (no diarization) plus a keyboard "end patient turn" key the operator presses off-screen. Invisible to judges, preserves the live-voice illusion. Typed *transcripts* stay banned; operator *cues* are allowed.
4. **H9–13:** Orchestrator tick (first task: **measure one real tick end-to-end** against the 3s budget) + station view (transcript pane, suggestion cards, guardrail toasts, contradiction cards).
5. **H13–15:** Chart entries + handoff view.
6. **H15–17:** Beat C — image upload → multimodal read → findings appended to `chart_entries`.
7. **H17–19:** Demo polish: seed tuning, nurse-persona prompt tuning against the rehearsal script, disclaimer banner.
8. **H19–21:** Two full rehearsals; record the backup video.
9. **H21–24:** Slack and sleep. This buffer absorbs overruns from every block above — it is not a feature window. Beat B (computer-use EHR) and a second guardrail scenario are explicitly **post-hackathon** ideas; three tired hours were never going to produce working computer-use.

## The Assignment

Before writing any code, write the **demo script verbatim** — every line the "doctor" and "patient" role-players will say, including the exact sentence that trips the guardrail ("it's been spreading to my left arm") and the exact sentence that plants the contradiction ("I've been taking the aspirin every day" after the chart says they stopped). The script is the spec: the seed data, the guardrails, and the prompts all get built to make that script land. Thirty minutes now, and it converts your 24 hours from "build a product" into "build a performance."
